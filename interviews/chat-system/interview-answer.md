## 1. Message Delivery Pipeline

### Architecture

```
Client A
  │
  ▼ WebSocket (persistent)
┌─────────────────────┐
│   WS Gateway (A)    │  ← horizontal pod; sticky connection per user
└──────────┬──────────┘
           │ produce (key = conversation_id)
           ▼
┌──────────────────────────────┐
│   Kafka: messages topic       │  ← partitioned by conversation_id
│   (durable, replayable)       │     guarantees ordered delivery per chat
└──────────┬───────────────────┘
           │ consume
           ▼
┌─────────────────────┐
│  Message Processor  │  ← persists to Cassandra, resolves recipients
└──────────┬──────────┘
           │ publish (channel = user_id)
           ▼
┌──────────────────────────────┐
│  Redis Pub/Sub               │  ← low-latency last-mile fan-out
│  (ephemeral, not durable)    │     WS gateways subscribe per connected user
└──────────┬───────────────────┘
           │
           ▼ WebSocket push
┌─────────────────────┐
│   WS Gateway (B)    │
└──────────┬──────────┘
           │
           ▼
        Client B
```

### Why Two Brokers?

| Layer | Technology | Role | Why Not Use the Other |
|---|---|---|---|
| Durable queue | Kafka | Ordered, replayable, fan-out to consumers | Too slow for real-time push (ms adds up) |
| Last-mile push | Redis Pub/Sub | Sub-millisecond publish to connected WS gateways | Not durable — message lost if subscriber disconnects |
| Alternative | Redis Streams | Durable + ordered per stream | Use instead of Kafka for simpler ops at moderate scale |

### At-Least-Once Delivery and Dedup

Every message carries a `client_message_id` (UUID generated by sender). The processor deduplicates on insert into Cassandra using `IF NOT EXISTS`. If the producer retries on Kafka write failure, the duplicate is silently dropped at storage.

```
Client sends: { client_message_id: "abc-123", content: "hi" }
Server stores: INSERT INTO messages (...) VALUES (...) IF NOT EXISTS
Duplicate retry: silently ignored — idempotent
```

> **Senior callout:** Partition Kafka by `conversation_id`, not `user_id`. This guarantees message ordering within a conversation while distributing load across partitions. Partitioning by user_id would mix messages from different chats into the same ordered sequence unnecessarily.

## 2. Storage Strategy

### Storage Layer Map

| Data Type | Technology | Partition / Index Strategy | Access Pattern |
|---|---|---|---|
| Message history | Cassandra | `partition_key = conversation_id`, `clustering_key = message_id DESC` | Latest N messages; paginate older by cursor |
| User profiles | PostgreSQL | Primary key = user_id; index on phone/email | Point lookup; rarely updated |
| Group metadata | PostgreSQL | Primary key = group_id; members in join table | Fan-out list, member count |
| Media files | S3 + CloudFront CDN | Key = `/{user_id}/{message_id}/{filename}` | Presigned URL per access; CDN for reads |
| Offline message queue | Cassandra / Redis | `partition_key = recipient_user_id` | Drain on reconnect; TTL 30 days |
| Notification tokens | PostgreSQL | Index on user_id | Lookup all device tokens for a user |
| Encryption key bundles | PostgreSQL | Index on user_id | Fetch recipient's public key bundle at send time |

### Chat History — Why Cassandra

```
Cassandra schema:
  CREATE TABLE messages (
    conversation_id UUID,
    message_id      TIMEUUID,        -- Snowflake/UUIDv7: time-ordered
    sender_id       UUID,
    content         BLOB,            -- encrypted ciphertext
    media_url       TEXT,
    status          TEXT,
    created_at      TIMESTAMP,
    PRIMARY KEY (conversation_id, message_id)
  ) WITH CLUSTERING ORDER BY (message_id DESC);
```

Queries are always `WHERE conversation_id = ? LIMIT N` — this maps perfectly to Cassandra's partition model. No joins needed. Write throughput scales horizontally.

### Media Storage Pipeline

```
Client selects file
  → generate thumbnail via Canvas API (client-side, instant preview)
  → POST /upload/initiate → server returns { upload_id, chunk_urls[] }
  → upload chunks to S3 presigned URLs (5MB each, 3 parallel)
  → POST /upload/complete → server triggers CDN cache warm + transcoding
  → send message with { media_url, thumbnail_url, mime_type, size }
```

### Notification Architecture

```
Message Processor
  │ produces to
  ▼
Kafka: notifications topic
  │ consumed by
  ▼
Notification Service
  ├─ user online → skip push (message already delivered via WS)
  ├─ user offline → lookup device tokens → APNs (iOS) / FCM (Android)
  └─ group notification → fanout: one Kafka message → N push payloads
```

> **Senior callout:** Never send push notifications synchronously in the message delivery path. The push will be delivered even if APNs/FCM is slow. Decouple via the Kafka notification topic so message delivery latency is unaffected by push provider latency.

## 3. Presence and Delivery / Read Acknowledgment

### Presence System

```
Client heartbeat every 30s → WS Gateway → Redis SET presence:{user_id} "online" EX 60

If heartbeat stops → key expires → user transitions to "offline"

Presence fan-out:
  On status change → Notification Service → Redis Pub/Sub channel per subscriber
  Subscribers: mutual contacts, group members currently viewing a chat with this user
```

| Event | Trigger | Delivery Mechanism | TTL / Expiry |
|---|---|---|---|
| Online | WS connection established | Redis SET + fan-out | 60s (refreshed by heartbeat) |
| Typing | Keydown + 300ms debounce | WS message → Redis SET EX 3s | 3s auto-expire |
| Offline | Heartbeat missed (TTL expires) | Redis keyevent → fan-out | Immediate on expiry |
| Last seen | Offline transition | Write to PostgreSQL | Persistent |

### Delivery Acknowledgment — Three-Tick Model

```
Sender client:   ✓   (sent — stored in Kafka/Cassandra)
Recipient WS:    ✓✓  (delivered — WS gateway received and pushed)
Recipient read:  ✓✓  (blue — recipient's client sent explicit READ ACK)
```

**Flow:**

```
1. Sender sends message → Kafka → Message Processor → Cassandra
   Processor responds to sender's WS: { event: "sent", message_id }         → ✓

2. WS Gateway pushes to recipient → recipient WS client receives
   Recipient gateway sends ACK upstream: { event: "delivered", message_id }  → ✓✓

3. Recipient opens conversation, scrolls to message (IntersectionObserver)
   Client sends: { event: "read", message_id, conversation_id }
   Server updates Cassandra: SET read_at = now() WHERE message_id = ?        → ✓✓ (blue)
```

**Offline delivery:** If recipient is offline, message is written to `offline_queue` in Cassandra. On reconnect, client sends `{ last_received_id }` and server drains the queue from that cursor.

> **Senior callout:** For group chats with 1,000+ members, tracking per-user read receipts per message is expensive. WhatsApp switches to showing only a count ("Read by 47") once group size exceeds ~20. At 100k+ members (Discord), only approximate counts via HyperLogLog are feasible.

## 4. End-to-End Encryption and Secure Storage

### Signal Protocol Overview

```
Key Exchange (X3DH — Extended Triple Diffie-Hellman):
  Alice fetches Bob's key bundle from server: { identity_key, signed_prekey, one_time_prekey }
  Alice computes shared secret using her keys + Bob's bundle
  No server involvement in key material — server only stores public key bundles

Session Encryption (Double Ratchet):
  Each message advances the ratchet → new encryption key per message
  Forward secrecy: compromise of current key doesn't expose past messages
  Break-in recovery: ratchet re-keys after every message exchange
```

### What the Server Stores vs What It Doesn't

| Data | Stored on Server | Notes |
|---|---|---|
| Ciphertext (message body) | ✅ Yes | Encrypted blob — server cannot read |
| Public key bundles | ✅ Yes | Used by senders to initiate sessions |
| Private keys | ❌ Never | Generated and stored client-side only |
| Plaintext messages | ❌ Never | Server has zero knowledge of content |
| Media files | ✅ Yes (encrypted) | Client encrypts before upload; key in message |

### Secure Local Storage

```
Local DB encryption (SQLite / Realm):
  Key = PBKDF2(user_password, salt, 100_000_iterations) → AES-256-GCM
  Salt stored in OS secure enclave / Keychain (iOS) / Android Keystore
  DB decrypted in-memory on app unlock; re-encrypted on background

Media key:
  Each media file encrypted with a random AES-256 key
  That key is encrypted with the session key and included in the message payload
  Server stores encrypted media; only recipients with the session key can decrypt
```

### E2EE Tradeoffs

| Capability | E2EE (Signal) | Server-Side Encryption |
|---|---|---|
| Server reads content | ❌ No | ✅ Yes |
| Backup / restore | Complex (key backup required) | Transparent |
| Content moderation | ❌ Not possible | ✅ Possible |
| Key loss = data loss | ✅ Yes | ❌ No |
| Compliance (GDPR erasure) | Deleting ciphertext is sufficient | Must delete plaintext |
| Multi-device sync | Requires key distribution (sender keys) | Transparent |

> **Senior callout:** For group E2EE at scale (Sender Keys, as used by WhatsApp/Signal): the sender encrypts the message once with a shared group session key, not once per recipient. This reduces encryption work from O(members) to O(1). The group key is rotated when a member leaves.

## 5. Real-Time Message Rendering Architecture

### Client State Flow

```
WebSocket message received
  │
  ▼
Normalize into store (by message_id)
  │
  ▼
Optimistic render (status = "pending")
  │
  ├─ Server ACK received → update status = "sent"   → ✓  tick
  ├─ Delivery event received → status = "delivered" → ✓✓ tick
  ├─ Read event received → status = "read"           → ✓✓ blue
  └─ Timeout / error → status = "failed"            → show retry UI
```

### Message State Machine

```ts
type MessageStatus = "pending" | "sent" | "delivered" | "read" | "failed";

interface Message {
  id: string;               // server-assigned (Snowflake / UUIDv7)
  clientId: string;         // UUID generated by sender before send
  conversationId: string;
  content: string;
  status: MessageStatus;
  sentAt: number;           // client timestamp (display)
  serverAt?: number;        // server timestamp (ordering)
}

// On send:
const msg: Message = { clientId: uuid(), status: "pending", ... };
store.addMessage(msg);
ws.send({ ...msg });

// On server ACK:
store.updateStatus(msg.clientId, "sent", { id: serverAssignedId });
```

### Virtual List for Performance

Rendering 50,000 messages in the DOM causes memory and layout thrash. Use windowed rendering:

```
react-window (or TanStack Virtual):
  Only DOM nodes for visible messages are mounted
  Off-screen messages are unmounted but their data stays in the store
  As user scrolls, unmount top, mount bottom (or reverse for upward scroll)

Result:
  DOM nodes: ~30 at any time (visible viewport)
  Store: all messages in memory (lightweight objects)
  Scroll performance: 60fps regardless of history length
```

| Approach | DOM Nodes at 50k messages | Scroll FPS | Memory |
|---|---|---|---|
| Naive render all | 50,000 | <10 fps | ~2GB |
| Virtual list (react-window) | ~30 | 60 fps | ~50MB |
| Pagination without virtual list | ~100 per page | 60 fps | Low, but loses scroll position |

> **Senior callout:** When prepending older messages (infinite scroll upward), the browser resets scroll position to the top. Use the CSS Scroll Anchoring API (`overflow-anchor: auto` on the container + an anchor element at the bottom) or save and restore `scrollTop` manually after the DOM update.

## 6. Read Receipts, Typing Indicators, and Live Status

### Typing Indicator — Client to Server

```ts
let typingTimeout: ReturnType<typeof setTimeout>;

function onKeyDown() {
  if (!isTyping) {
    ws.send({ event: "typing_start", conversationId });
    isTyping = true;
  }
  clearTimeout(typingTimeout);
  typingTimeout = setTimeout(() => {
    ws.send({ event: "typing_stop", conversationId });
    isTyping = false;
  }, 2000);  // stop signal if no keypress for 2s
}
```

**Server:** stores `typing:{conversation_id}:{user_id}` in Redis with TTL 3s. Broadcasts to other participants via Redis Pub/Sub. If the key expires (user closed app without stop event), the indicator auto-clears.

### Read Receipts — Batched ACK

```ts
const readQueue = new Set<string>();  // message_ids seen

const observer = new IntersectionObserver((entries) => {
  entries.forEach(e => {
    if (e.isIntersecting) readQueue.add(e.target.dataset.messageId!);
  });
}, { threshold: 1.0 });

// Flush every 500ms — not on every scroll event
setInterval(() => {
  if (readQueue.size === 0) return;
  ws.send({ event: "read_ack", messageIds: [...readQueue], conversationId });
  readQueue.clear();
}, 500);
```

### Event Frequency Budget

| UI Event | Client Sends | Server Receives | Debounce / Throttle |
|---|---|---|---|
| Typing start/stop | Per keystroke pause | ~1 event/2s per user | Client debounce 300ms + 2s idle timeout |
| Read ACK | Per visible message | Batched every 500ms | Client batch flush |
| Presence heartbeat | Every 30s | 30s interval | Fixed interval |
| Online status push | On WS connect/disconnect | Event-driven | No throttle needed |
| Message delivery ACK | Per message received | Per message | No throttle needed |

### Presence Badge

```
Primary: WebSocket push → instant update
Fallback (if WS drops): poll GET /presence/{user_id} every 60s

Badge logic:
  "online"  → seen heartbeat in last 60s
  "away"    → app backgrounded > 5 min (mobile sends away event on background)
  "offline" → Redis TTL expired (no heartbeat for 60s)
  "last seen X ago" → timestamp from PostgreSQL on offline transition
```

> **Senior callout:** Mutual presence is expensive at scale. WhatsApp only broadcasts your presence to users in your contact list who are currently online and have your chat open — not to all 2B users. Lazy fan-out: subscribe to presence only when the chat UI is active.

## 7. Chat History Pagination and Infinite Scroll

### Cursor-Based Pagination (Not Offset)

```
Wrong — offset-based:
  GET /messages?conversation_id=X&offset=100&limit=20
  Problem: if a new message is inserted, offset shifts → you skip or repeat messages

Correct — cursor-based:
  GET /messages?conversation_id=X&before_message_id=MSG-789&limit=20
  Server: SELECT * FROM messages WHERE conversation_id = X
            AND message_id < MSG-789
            ORDER BY message_id DESC LIMIT 20
  Returns: messages + { next_cursor: MSG-567, has_more: true }
```

### Infinite Scroll Implementation

```ts
const sentinelRef = useRef<HTMLDivElement>(null);

useEffect(() => {
  const observer = new IntersectionObserver(([entry]) => {
    if (entry.isIntersecting && hasMore && !loading) {
      fetchOlderMessages(oldestMessageId);
    }
  }, { rootMargin: "200px" });  // trigger 200px before sentinel is visible

  if (sentinelRef.current) observer.observe(sentinelRef.current);
  return () => observer.disconnect();
}, [hasMore, loading, oldestMessageId]);

// In JSX:
// <div ref={sentinelRef} />  ← placed at the top of the message list
// Messages render below it in reverse order
```

### Scroll Position Preservation

```ts
// Before prepending older messages:
const container = listRef.current;
const prevScrollHeight = container.scrollHeight;
const prevScrollTop = container.scrollTop;

// After DOM update (older messages prepended at top):
const newScrollHeight = container.scrollHeight;
container.scrollTop = prevScrollTop + (newScrollHeight - prevScrollHeight);
// User stays at the same visual position — no jump
```

| Concern | Solution |
|---|---|
| Scroll jumps when prepending | Save/restore `scrollTop` delta after prepend |
| DOM memory at 50k messages | Virtual windowing — evict off-screen nodes |
| Offline history | IndexedDB as local cache; serve from cache if network unavailable |
| Duplicate messages on reconnect | Dedup by `message_id` in the store before rendering |

> **Senior callout:** The browser's native Scroll Anchoring (`overflow-anchor: auto`) works for most prepend cases but fails in some mobile WebViews. For production, implement manual `scrollTop` delta adjustment as above — it's reliable across all environments.

## 8. File Upload — Previews, Retries, Progressive Rendering

### Full Upload Flow

```
User selects file (image / video / document)
  │
  ├─ [Immediate] Generate thumbnail client-side
  │    Image: Canvas API → drawImage → toBlob → ObjectURL
  │    Video: load first frame via <video> + seek(0) → Canvas capture
  │    Show preview instantly — no network required
  │
  ├─ POST /upload/initiate { filename, size, mime_type }
  │    Server responds: { upload_id, chunk_urls: [presigned_s3_url × N] }
  │
  ├─ Split file into 5MB chunks
  │    Upload chunks in parallel (max 3 concurrent) via PUT to presigned URLs
  │    Track progress per chunk → aggregate into single progress bar
  │
  ├─ On chunk failure:
  │    Retry up to 3× with exponential backoff (1s, 2s, 4s)
  │    Persist chunk manifest to localStorage: { upload_id, completed_chunks[] }
  │    On page refresh → resume from last completed chunk
  │
  └─ POST /upload/complete { upload_id, chunk_etags[] }
       Server: S3 CompleteMultipartUpload → CDN cache warm → trigger transcoder
       Responds: { media_url, thumbnail_url }

Send message: { content, media_url, thumbnail_url, mime_type, file_size }
```

### Chunk Upload with Retry

```ts
async function uploadChunk(url: string, chunk: Blob, attempt = 0): Promise<string> {
  try {
    const res = await fetch(url, { method: "PUT", body: chunk });
    return res.headers.get("ETag")!;
  } catch (err) {
    if (attempt >= 3) throw err;
    await sleep(1000 * 2 ** attempt);  // 1s, 2s, 4s
    return uploadChunk(url, chunk, attempt + 1);
  }
}

async function uploadFile(file: File, chunkUrls: string[]): Promise<string[]> {
  const CHUNK_SIZE = 5 * 1024 * 1024;
  const chunks = Array.from({ length: chunkUrls.length }, (_, i) =>
    file.slice(i * CHUNK_SIZE, (i + 1) * CHUNK_SIZE)
  );

  // 3 concurrent uploads
  const etags: string[] = new Array(chunks.length);
  const queue = chunks.map((chunk, i) => ({ chunk, url: chunkUrls[i], i }));

  await Promise.all(
    Array.from({ length: 3 }, async () => {
      while (queue.length > 0) {
        const { chunk, url, i } = queue.shift()!;
        etags[i] = await uploadChunk(url, chunk);
        updateProgress(i);
      }
    })
  );

  return etags;
}
```

### Progressive Rendering

| File Type | Before Upload Completes | After Upload Completes |
|---|---|---|
| Image | BlurHash placeholder + client-side ObjectURL preview | Full-resolution from CDN |
| Video | First-frame thumbnail from Canvas + spinner | Video player with CDN URL + poster |
| Document | File icon + filename + size | Download link |
| Large image | LQIP (Low Quality Image Placeholder) → progressive JPEG | Full-res lazy-loaded |

```
Recipient message rendering:
  Receives message with { thumbnail_url, media_url, blurhash }
  1. Render BlurHash placeholder immediately (pure CSS — no network)
  2. Load thumbnail_url (small, fast CDN hit) — replace placeholder
  3. On click / in viewport → lazy-load full media_url
```

> **Senior callout:** Never block the send button waiting for upload to complete. Send the message with `status = "uploading"` immediately after thumbnail generation. Upload in the background. On completion, update the message with the real `media_url`. This is how iMessage, WhatsApp, and Telegram achieve instant-feeling sends even for large files.

## Quick Reference

| Topic | Key Decision | Why |
|---|---|---|
| Message broker | Kafka (durable) + Redis Pub/Sub (last-mile) | Kafka for ordering/replay; Redis for sub-ms push |
| Chat storage | Cassandra, partition by conversation_id | High write throughput; optimal for latest-N queries |
| Media storage | S3 + CDN, multipart upload | Durable at scale; CDN for read throughput |
| Presence | Redis key + 60s TTL + heartbeat every 30s | Auto-expire on disconnect without explicit offline signal |
| Read receipt | IntersectionObserver + 500ms batch ACK | Accurate without flooding server on every scroll |
| Typing indicator | Client debounce 300ms + Redis TTL 3s | Auto-clears if app crashes; no explicit stop needed |
| Pagination | Cursor-based (before_message_id) | No skip/repeat on concurrent inserts |
| Infinite scroll | IntersectionObserver on top sentinel | Efficient; no scroll event listeners |
| Scroll preservation | scrollTop delta save/restore | Reliable across browsers including mobile WebViews |
| File upload | Chunked 5MB + 3 parallel + localStorage manifest | Resumable; progress feedback; no full re-upload on failure |
| E2EE | Signal Protocol (X3DH + Double Ratchet) | Forward secrecy; zero server knowledge |
| Optimistic UI | Render pending → server ACK → commit/rollback | Instant feel; correct on failure |
| Virtual list | react-window / TanStack Virtual | 60fps at 50k+ messages; bounded DOM node count |
| Group read receipts | Exact count < 20 members; approximate (HLL) beyond | Per-member tracking at 100k members is infeasible |
| Push notification | Async via Kafka notification topic | Decouples delivery latency from push provider latency |
